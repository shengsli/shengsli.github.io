1 functional testing

  • functional testing derives test cases from program specifications
  • aka:
    • specification-based testing (from specifications)
    • black-box testing (no view of the code)
  • functional specification = description of intended program behaviour (formal or informal)

2 systematic vs random testing

  1. random (uniform):
    • pick possible inputs uniformly
    • avoid designer bias
      • the test designer can make the same logical mistakes.
    • treats all inputs equally valuable
  2. systematic (non-uniform):
    • try to select inputs that are especially valuable
    • usually by choosing representatives of classes (that are apt to fail often or note at all).
  3. functional testing is systematic testing.

3 why not random?

  • non-uniform distribution of faults

4 consider the purpose of testing

  • look systematically/non-uniformly

5 partition principle

  • samples that are more likely to include special or trouble-prone regions of the input space
  • quasi- partition testing: separate the input space into classes whose union is the entire space
  • desirable case: each fault leads to failures that are dense in some class of inputs

6 functional testing: exploiting the specification

  • functional testing uses the specification to partition the input space
  • test each category. and boundaries between categories

7 why functional testing? more on p10

  • the baseline for designing test cases
  • timely
  • effective
  • widely applicable
  • economical

8 early functional test design

  • program code is not necessary
    • a description of intended behaviour is nedded
  • early functional test design has side benefits
    • reveal ambiguities and inconsistency in spec

9 functional vs. structural

9.1

  • functional
  • structural
  • fault-based
  • model-based
  • functional testing is best for missing logic faults
  • structural testing never focus on code that is not there

9.2 granularity levels

  • functional test applies at all granularity levels
    • unit: module interface
    • integration: api or subsystem
    • system: system requirements
    • regression: system requirements + bug history
  • structural test applies to unit and integration

10 steps: from spec to test cases

  1. decompose the spec, break it into testable features
  2. select representatives
  3. form test specifications = representative values or model behaviours
  4. produce and execute actual tests

11 summary

  • functional testing: generation of test cases from specifications is valuable and flexible approach to software testing
  • systematic testing is intentionally non-uniform to address special cases, error conditions, etc.
