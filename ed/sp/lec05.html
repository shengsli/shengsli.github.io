2 countermeasures

  • 2 basic programming-related countermeasures:
    1. treat the symptoms:
      • limit damage that can be done by attacks
      • containment and curtailment
    2. treat the cause:
      • ensure code contain no vulnerability
      • code review, analysis
  • why choice 2 is impossible?
    • deployed on too many machines
    • original author is not available or masterful enough to fix

2.1 generic defence

  • defensive technologies
    • gives defence in depth against new attacks, malware
    • code replacement is expensive
    • code cannot be replaced
  • e.g. upgradeable firmware

2.2 defences against overflows

2.2.1 tamper detection in software

  1. stack canary
    • canary sits below return address
    • defeat stack canary
      • predict where canary is, and write an exact canary value
      • hop over canary
      • defeat canary mechanism
    • what happens if an overflow is detected?
      • stop program and exit with warning
      • maybe raise an alarm
      • maybe rolling back to some point
    • GCC's stack smashing protector 
    #include <stdio.h>
#include <string.h>

int fun1 (char *arg)
{
	charbuffer[1024];
	strcpy(buffer, arg);
}

void main (int argc, char *argv[])
{
	fun1(argv[1]);
}
  2. security arms race
    • attack: probe canary
    • defence: pseudo-random sequence, cryptographic PRNG
  3. stack canary effectiveness
    • heap implementation
    • weaknesses:
      1. does not protect against local variable overwriting
        • related mechanisms reorder local variables
      2. other attacks work by overwriting parameters
        • aim to change where subsequent writes occur
        • overwrite return address, but do not return
    • hardened heap implementations have also been developed
      • glibc and windows since XP SP2 have heap canaries
      • but application specific heaps, HLLs not covered
  4. better attacks, better detection
    • return-to-libc, and return-oriented programming (ROP)
      • state-of-the-art: use existing executable code
      • evades canaries, also defeats NX

    A more powerful and defence mechanism is Control-Flow Integrity, which ensures that code execution follows a pre-determined call graph. This can defend against ROP and similar attacks, depending on the accuracy and granularity of the enforcement.

2.2.2 memory protection in software

  1. operating system separation
    • isolation
    • sharing
    • concern: granularity of protection
  2. hardware memory protection mechanisms
    • fences: separate memory accesses between OS and user code mode
    • base and bounds registers
    • tagged architecture
  3. memory separation: segmentation & paging
    • segmentation
    • paging
    • some OSes goes back to flat memory model
  4. non-executable memory pages
    • CPUs have often included R, RW, X protection for memory pages.

2.2.3 diversification methods

  1. addressing space layout randomization (ASLR)
    • concept: use diversification to make many versions of same program
    • thwarts general attacks that make assumptions about fixed structure.
    • ASLR: make it harder to find data ot code locations, by randomising layout during load time. Breks hard-coded static locations.
    • implemented in Linux by the PaX Team.
