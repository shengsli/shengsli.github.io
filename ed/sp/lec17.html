2 man at the end attacks (MATE)

2.1 inside vs. outside threats

  • inside: security perimeter
  • outside: exploit digital assets

2.2 MATE and R-MATE

  • MATE attacks: occur where an adversary has physical access to a device and compromises it by inspecting, reverse or tampering with its hardware or software
  • remote MATE attacks: occur in distributed systems where untrusted clients are in frequent communication with trusted server over a network, and where a malicious user can get an advantage by compromising an untrusted device. e.g. blockchain

2.3 MATE attack scenarios

  1. software piracy
  2. license check removal
    • defence: make program tamperproof
  3. malicious reverse engineering
    • attack: "code lifting", i.e. copy program into attacker's own program
    • defence: obfuscate code to make reverse engineering difficult
  4. digital rights management (DRM) key extraction
  5. protocol discovery
  6. violation of export/supply chain controls
  • Mostly, these attach non-availability where there is an attempt to deny access to an asset.

2.4 R-MATE attack scenarios

  1. Cheating in networked computer games
    • Bob reuses cashed data
    • defence: replay-resilient protocol (key defence), tamperproofing
  2. assessing or altering distributed medical records
    • e.g. Alice collects data from a wireless sensor network.
    • Bob interferes with some of the sensors.
    • Bob can swap data or mess up sensors' reading.
      • hacking smart meters to disrupt supply
    • defence: anomaly detection, remote attestation, machine learning, more fuzzy type techniques

3 code signing

  • verify if it comes from trusted
  • integrity and authenticity, not confidentiality
  • trusted platform
    • tamperproof
    • reverse engineering: encryption -> confidentiality, attacker cannot see code
      • stop people from looking the code

4 program obfuscation

  • write code in a difficult way to understand
  • there are many contests, like international obfuscated c code contest

4.1 obfuscating compilation

  • transform a program p into a functionally equivalent program p'
  • properties of a obfuscating compiler
    • make it hard to reverse from p' to p

4.2 practical obfuscation

  • obfuscating techniques
    1. rename identifiers: sensible names -> confusing names
    2. equivalent expressions, arithmetic transformation, e.g. bit shift multiply. but compiler may optimise it.
    3. reorder, duplicate, add dummies -> make it hard to understand and might disorder reverse engineering. again, compiler
    4. flatten control flow. e.g.
      • unroll loop
      • conditional -> jump table
      • again, compiler
    5. merge and split functions. inline, outline
    6. introduce pointer aliases. add concurrency. (more advanced)
      • race condition
    7. opaque predicates
    8. custom abstract machine
  • when use these tools, examine how rigorous they are

4.3 obfuscation in theory

  • black box input-output behaviour
  • virtual black-box security

4.4 power of obfuscation

  • powerful building block for cryptographic primitives
  • symmetric and asymmetric crypto
  • homographic encryption -cloud computing: enable without sharing my data
