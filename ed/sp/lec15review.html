1 why C can be trickier than I and A?

  • all 3 access control

2 single origin restriction

  • single origin policy (SOP): web page elements must come from same domain.
  • no mashups
    • mashups: online applications that combine data and tools from different websites.
    • mixing things from different places to one place is risky.

3 generalised origin-based restrictions

  • allow mashups but control things better (e.g. avoid XSS)
  • web page loads script content from many places
  • 2 security features of a browser:
    1. CSP: content security policy (whitelist sources)
      • particularly relevant to executing scripts. prevent XSS
    2. CORS: cross-origin resource sharing (permit CO requests)
      • e.g. Ajax request from one domain to another domain. (from a legitimate domain to another)
  • access control:
    1. discretionary access control: access control can be set individually by owner of the file.
      • but cannot prevent owner's accidental mistake and insider threats.
    2. mandatory access control: system-wide access control

4 a solution: separate confidential from non-confidential data

7 language-based security

7.1 what is it?

  • It offers a model of security based closely to the applications.
    • i.e. It prevents applications-level attacks inside the application.
  • examples of concepts that we want to protect, like:
    1. when manipulating users: e.g. social messaging applications
      • we have concepts like prove who the users are
    2. in network traffic: things are missed and lost. It's not clear what we are trying to protect.

8 taint tracking: an example of language-based security for information flow

  • a dynamic technique
  • idea: attach labels to data inputs (sources) and data outputs (sinks)
    • propagate labels during computation (cf. dynamic typing)
    • data comes from trusted sources is untainted.
    • data from taint sources, like user input, is tainted.
