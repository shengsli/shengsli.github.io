3 sequential memory consistency model (SC)

3.1 rules:

  1. ordering of atomic actions (particularly reads and writes to memory) from any one thread have to occur in normal program order.
  2. atomic actions from different threads are interleaved arbitrarily (i.e. in an unpredictable sequential order subject only to rule 1) with every thread seeing this same ordering.

3.2 model

  • analogy: think of SC execution as a random switch, allowing processes to access memory once at a time.

4 atomic actions

  • e.g. b = a + a. A sensible compiler would implement it with one read of a.
  • notation: <...> is atomic.

5 atomic actions and interleavings

co
    count++; // count++;
oc
  • 3 actions: read count; increment 1; write count;
  • Question: How many permutations? 20
    • Some math: \begin{align} \frac{(N_a+N_b)!}{N_a!N_b!} &= \frac{(3+3)!}{3!3!} &= 20 \end{align} 
co
    <count++;> // <count++;>
oc

5.1 example (error cases): This program is to reverse order by swapping.

5.1.1 case 1

co [i = 0 to n-1] {
    a[i] = a[n-i-1];    ## try to reverse a in parallel
}
  • It has chance to be right. Both threads read first and then write to location.

5.1.2 case 2

co [i = 0 to n-1] {
    <a[i] = a[n-i-1];>  ## try to reverse a in parallel
}
  • It's worse. It's definitely not going to give the expected result.
  • It will lose target data after writing to target. It needs a temporary variable.

6 await notation

  • It captures the synchronization behaviour.
  • It is really hard to implement.
  • < await (B) S >
    • B: Boolean/condition
    • S: Statements/action
    • S happens only if B is true.
  • example 1: 
a = 0; flag = 0;
co
  {a=25; flag=1;}
//
  <await (flag==1) x=a;>
oc
  • example 2: 
a = 0; b = 0;
co 
  {a=1; other stuff; a=0;}
//
  <await (a==1) b=a;>
  • Possible results:
    1. a = 0; b =1;
    2. This program may hang (fail to terminate).
