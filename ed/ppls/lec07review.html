1 Big picture

  • Today we are gonna change our implementation style.
  • The mechanisms we have designed so far all have been implemented directly in the user-address space.
    • probably hidden inside a library, but nonetheless invisible to the OS.
    • e.g. spinning version of lock implementation
    • e.g. looping in barrier implementation
  • drawback:
    • depend on the system that the program is running on and what is running at the same time
    • when threads are waiting. they are taking up CPU cycles, e.g. reads when waiting lock changes.
  • We will look at techniques which avoid resource wastage problem by having help of OS.
    • e.g. OS suspends and reactivates/resumes threads
    • provide the same logical behaviours like lock and barrier without overhead of spinning

5 producers and consumers

  • scenario: multiple producers and multiple consumers

    • wait if there is no job
  • how many producers and consumers?

6 version 1:

  1. sem full: counts how many space of buffer is full
  2. sem empty: counts how many space of buffer is empty
  3. buffer
  • full and empty count resources
  • requirement of stopping producers is based on # of empty spaces
  • requirement of stopping consumers is based on # of full spaces
  • producers carry on when empty is non-negative
  • consumers carry on when full is non-negative
