6 critical sections

  • NOTE: atomicity is equivalent to critical section.
    • In this context (p45), atomicity is a bit of overkill. Atomicity is a stronger requirement.
    • The difference is that if we make it atomic, no one can see it as intermediate states.
co [i = 1 to n] {
  while (something) {
    critical section;      ## one thread at a time
    non-critical section;
  }
}

6.1 important properties

  1. mutual exclusion
  2. absence of deadlock (or livelock)
  3. absence of unnecessary delay
  4. eventual entry (or no starvation)
  • Remarks: the fist three are always essential. eventual entry may note matter in some performance parallel programs, as long as we are making progress elsewhere.

6.2 critical sections and locks

  • locking and unlocking

6.3 implementing locks

  • A simple approach is to implement each lock with a shared boolean variable.
    • lock = false when it is free.
lock_t l = false;
co [i = 1 to n] {
  while (something) {
    < await (!l) l = true; > ## we need this await statement 
                             ## to guarantee only one winner enter c.s.
    critical section;
    l = false;
    non-critical section;
  }
}
  • (recall that our model assumes that the l = false; write is already atomic.)

6.4 wrong implementation of critical section using locking

while (l);
l = true;
l = false;
  • 2 threads can enter the critical section at the same time.

6.5 low-level perspective: atomic instruction

  • There are many possibilities, including
    • fetch-and-add,
    • test-and-set (TS),
    • load-linked,
    • store-test and set instruction
  • NOTE: we think of TS as a call-by-reference function. But it is a single machine instruction in reality.
bool TS (bool v) {
  < bool initial = v;
    v = true;
    return initial; >
}
  • The key feature is that this happens or at least, appears to happen atomically.

6.6 solution: implementing locks using TS

6.6.1 version 1

lock_t l = false;
co [i = 1 to n] {
  while (something) {
    while (TS(1));  ## i.e. spin
    critical section;
    non-critical section;
  }
}
  • All cache coherence systems see it as a write. We don't want many writes to shared variables.
  • Simple spin locks don't make good use of the cache. TSs play havoc with contention and coherence performance.

6.6.2 version 2: refinement, efficiency improvement

  • test-and-set
  • only try when you might have a chance
lock_t l = false;
co [i = 1 to n] {
  while (something) {
    do {
      while (l);      ## spin until lock seems free, read only
    } while (TS(l));  ## actual atomic locking
    critical section;
    l = false;
    non-critical section;
  }
}

6.6.3 version 3: concise form of version 2 (the same as version 2)

  • test-and-test-and-set
lock_t l = false;
co [i = 1 to n] {
  while (something) {
    while (l || TS(l)); 
    critical section;
    l = false;
    non-critical section;
  }
}
