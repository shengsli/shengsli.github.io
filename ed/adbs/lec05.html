1 Sorted files and binary search

  • Range query: 
    SELECT * FROM Customer 
 WHERE zipcode BETWEEN 8800 AND 8999
    • search phase: binary search find the first qualifying record \(\Rightarrow O(log_2 N)\), N records
    • scan phase: sequential access
  • Disadvantages:
    • additional disk I/O
    • Binary search jumps left or right. Unpredictable jumps is bad for page prefetching.

2.1 ISAM: Indexed Sequential Access Method p6

  • non-leaf pages:
    • A non-leaf page has multiple index entries.
    • An index entry is <search key value, page ID>.
    • Non-leaf pages only direct searches.
  • leaf pages:
    • contain index data entries k*. A data entry is <k, rid>.
    • primary pages
    • overflow pages: unordered
  • Leaf pages allocated sequentially
    • search: start at root, use key comparisons to go to leaf
    • insert:
    • delete

2.2 Comments on ISAM p11

  • non-leaf levels are not affected by inserts/deletes
    • need not be locked during concurrent index accesses
    • locking can be a bottleneck in dynamic tree indexes (particularly near the root)
  • ISAM may lose balance after heavy updating
    • create long chains of overflow pages
    • search performance can degrade over time
    • leaving free space ~20% during index creation partially reduces this problem

2.3 ISAM vs. Binary seach p12

  • N = # pages in the data file (search space)
  • \(F = \frac{max\ \# childern}{index node}\), \(F = 1000\) typically

The average number of children for a non-leaf node is called the fanout of the tree. If every non-leaf node has n children, a tree of height h has \(n^h\) leaf pages. In practice, nodes do not have the same number of children, but using the average value F for n, we still get a good approximation to the number of leaf pages, \(F^h\). (textbook)

  • After s steps down the tree, search space is reduced to \(N/F^s\)
    • how many steps? \(s = log_F N, F = 1000 typically\)
    • \(s = log_F N << log_2 N\)
    • ISAM is much more efficient than binary search

2.4 B+ Tree (most widely used index) p13

  • no overflow chains
  • remain balanced i.e. every leaf is at same depth
  • search performance only dependent on the height
    • height rarely exceeds 3 due to high fanout
  • efficient insert/delete procedures
    • data file grow/shrink dynamically, non-leaf nodes are modified
  • each node (except the root) has a minimum occupancy of 50%
    • each non-root node contains \(d \leq m \leq 2d\) entries, d is the order of the tree
  • leaf pages are doubly linked
  • side notes:
    • internal nodes aka index nodes

2.4.1 Examples p14

  1. calculation
    • Given
      • height h
      • order d
      • fill-factor
    • F = 2*d*fill-factor
    • capacity of B+ tree with height h = \(F^h\)
    • buffer pool
    level #page size
    1 1 8KB
    2 \(F\) 1MB
    3 \(F^2\) 138MB
  2. insert p16
    1. no redistribution strategy
      1. find leaf node L
      2. put data entry into L in sorted order
      3. if L has enough space, done
      4. else split L into L and L2
        • redistribute entries evenly, copy up middle key
        • insert index entry pointing to L2 into parent of L
      5. split inner node, redistribute entries evenly, push up middle key
    2. redistribution strategy
  3. delete p20
    1. find leaf nodelse e L
    2. remove the entry
    3. if L is at least half-full, done
    4. if L has only d-1 entries
      • redistribute, push through
      • or merge L, pull down
