1 Recap sec9.5

  • DBMS stores a database as a file (or more files).
  • a file is a collection of pages (or a collection of records).
  • a page is a collection of slots.
  • each slots contains a record.

2 Context

  • when dealing with I/O issue, we use page abstraction.
  • in higher levels of DBMS, we seen data as a collection of records.
  • In this section, we consider how a collection of records can be arranged on a page.
    • think of a page as a collection of slots
    • each slot contains a record
    • 2 ways to identify records:
      • use <page id, slot no.> as rid
      • alternatively, assign each record a unique integer as its rid
        • and maintain a table that lists the page and slot of the corresponding record for each id
      • remarks: maintaining the table introduces overhead, <page id, slot no.> is commonly used.
  • We now consider some alternative approaches to managing slots on a page.
    • main considerations are how these approaches support operations such as search, insert, delete records on a page.

3 Fixed-length Records

  • uniform record slots
  • record slots can be arranged consecutively within a page
  • insert:
    1. locate an empty slot
    2. place the record there
  • main issues are how we keep track of empty slots and how we locate all records on a page.
  • alternatives hinge on how we handle deletion of a record.
    • side notes: alternatives below handle record deletion in different ways.

3.1 alternative 1

  • N: # of records on the page
  • store records in the first N slots
  • whenever delete(r), move the last record on the page into the vacated slot.
  • remarks:
    • this format allows use to locate the i-th record on a page by a offset calculation.
    • however, this approach does not work if there are external references to the record that is moved.
      • because the rid contains the slot no., which is not changed.

3.2 alternative 2

  • handle deletions by using an array of bits
  • 1 bit per slot, keep track of free slot information
  • locate records by scanning the bit array to find slots whose bit is on
  • delete:
    • turn off deleted record's bit

3.3 extract from lec03 p11

  • strawman idea: keep track of # of records in a page and just append a new record to the end
  • delete:
    • move the last record to the empties slot
    • this changes the last record's id

4 Variable-length Record

4.1 temptation: why use slotted page organization?

  • it's the most flexible organization for variable-length records
  • it allows us to move records around on page
    • so that we can keep track of space freed by deletions
    • and keep records on a page sorted according to the value in some field.
  • problem: when a new record is to be inserted, we have to find an empty slot, we have to find an empty slot of just the right size.
    • a slot is too big: waste space
    • a slot is too small: cannot fit
    • when delete a record, we must move records to fill the hole created by deletion,
      • to ensure that all the free space on the page is contiguous.

  1. maintain a directory of slots for each page, with a <record offset, record length> per slot.
    • record offset: is a pointer to the record.
      • it is an offset in bytes from start of data area on the page to the start of the record.
    • delete: set the record offset to -1
    • records can be moved around on the page because rid
      • rid is the page number and slot number.
      • rid does not change when the record is moved; only the record offset in the slot changes.
  2. be careful with managing free space for new records
    • page is not pre-formatted into slots.
    • implementation: maintain a pointer, i.e. a offset from the start of data area on the page
      • indicate the start of the free space
      • when a new record is too large to fit, move records on the page to reclaim space free by deletion earlier.
        • the idea is to ensure that after reorganization, all records appear in contiguous order, followed by free space.
  3. note: slot for a deleted record cannot always be removed from the slot directory,
    • because slot numbers are used to identify records.
    • by deleting a slot, we decrement the slot number,
    • and thereby change rid of records pointed to by subsequent slots.
    • the only way to remove slots from the slot directory is to remove the last slot if the record that it points to is deleted.
    • when insert(record), we
      1. scan slot directory for a slot that does not point to any record,
      2. and this slot should be used for the new record.
      3. add a new slot to slot directory only if all existing slots point to records
    • inserts are more much common than deletes, typically.
      • thus, # of entries in the slot directory is likely to be very close to the actual # of records on the page
  4. slotted page organization is also useful for fixed-length records.
    • move records around frequently
      • e.g. maintain records in sorted order
    • store record length in system catalogue
  5. In some situations e.g. internal pages of B+ tree, (more in sec10)
    • we may not care about changing of a rid.
      • the slot directory can be compared after every record deletion
      • this strategy guarantees that # entries in slot directory is the same as # records on the page.
    • if we do not care modifying rids,
      • we can also sort records on a page by moving slot entries (rather than actual records).
  6. a simple variation on slotted page organization
    • maintain only record offsets in the slots
    • length is stored with record e.g. in 1st bytes
    • same slot directory structure as pages with fixed-length records

5 Remarks:

  • rid: <page id, slot no.>
  • slot: <record offset, record length>
